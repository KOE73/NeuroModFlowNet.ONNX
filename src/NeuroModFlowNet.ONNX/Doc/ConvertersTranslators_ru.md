# Трансляторы и конвертеры в NeuroModFlowNet.ONNX


- **Converter** — это менеджер, который знает, где взять память и куда её положить.

Конвертеры могут преобразовывать данные, но не должны заниматься оптимизацией. 
 Если есть максимально простой способ отобразить/скопировать байты из входаных данных
 в выходной буфер, то конвертер делает это сам. 
 
 В идеале если происходит простое отображение (mapping) без копирования, то конвертер должен его использовать.

- **Translator** — это высокопроизводительные функции, которые выполняют конкретные 
 трансформации данных максимально эффективно, используя все доступные ресурсы (SIMD, Unsafe, Zero-Allocation).
 
_Подсистема выросла из Трансляторов. Возможно это атавизм._ В самом начале были только трансляторы, 
 которые выполняли всю работу по подготовке данных. И надо было реально преобразовывать BGR-RGB, менять порядок каналов, нормализовать и т.д.
 После модификации модели, которая стала принимать данные в формате byte BGR, вся работа свелась на указать где исходные
 данные в памяти лежат. 
  


# Converters

Конвертер знает что он работает с конкретной моделью (OnnxModel) и конкретными входными данными (TIn).
Он знает:
 - как получить место в памяти, т.е. буфер из модели, 
    или напрямую замапить во входные параметры модели `OrtValue.CreateTensorValueWithData`.
 - как вызвать транслятор (Input2BufTranslator) для преобразования входных данных в нужный формат.

**ConverterBuilder** — это фабрика для создания высокопроизводительного делегата (InputConverter), 
 который подготавливает данные для нейросети.

## Зачем это нужно (Technical Rationale):
- **Устранение ветвлений в Hot Path**: В цикле обработки видео (инференс) нельзя каждый раз проверять: "А мы сейчас на CPU или GPU? А вход FP16 или FP32? А батч 1 или 4?". ConverterBuilder принимает эти решения один раз (в момент .Build()) и "запекает" выбранный алгоритм конвертации внутрь возвращаемого делегата. В итоге, во время работы (Predict) происходит прямой вызов нужного кода без if/switch.
- **Разделение ответственности (Decoupling)**:
  - Модель (OnnxModel) знает только про свои буферы (тензоры).
  - Входные данные (TIn) (например, List<Mat>) ничего не знают про ONNX.
   - Трансляторы (TTranslators) — это набор статических методов ("стратегий"), которые умеют превращать TIn -> Span<TOut>.
Builder связывает их вместе, скрывая сложность выбора конкретного метода трансляции.
Generic-магия для Type Safety: Класс жестко типизирован (TIn, TOut), чтобы компилятор не дал скормить float модели, ожидающей Float16, или Mat туда, где нужен byte[].

## Как это работает (Step-by-Step)
Конфигурация (WithAlgorithm): Ты говоришь билдеру: "Хочу использовать вот этот конкретный метод трансляции" (например, Positive_OpenCV_Blob). Если не сказал — сработает AutoTune (попытка угадать лучший алгоритм под железо/размер батча).

Сборка (Build):
1. Билдер берет выбранный низкоуровневый транслятор (Input2BufTranslator).
1. Билдер берет переданный ему "шаблон" конвертера (ConverterForTranslator), который знает, как достать буфер из модели.
1. Ключевой момент: Он создает замыкание (closure), объединяя их.
1. Результат кешируется в _CachedInputConverter.
1. Использование: Внешний код получает делегат InputConverter. При его вызове сразу выполняется нужный код копирования памяти.



# Converters Описание V2

## Этап 1: Подготовка (Cold Path — выполняется один раз)
Этот этап инкапсулирован в методах WithAlgorithm и Build. 
Его задача — убрать всю «умственную работу» из цикла обработки кадров.

1. **Выбор стратегии (Selection)**: Ты выбираешь конкретный математический алгоритм (Input2BufTranslator). Это может быть либо ручной выбор (например, «использовать OpenCV Blob»), либо автоматический через AutoTune, который может замерить скорость разных вариантов на текущем железе.
1. **Проверки (Validation)**: На этапе компиляции дженерики гарантируют, что типы данных (TIn, TOut) совместимы. На этапе выполнения Build проверяет, нашелся ли подходящий транслятор, чтобы не упасть с ошибкой позже.
1. **Запекание делегата (Closure/Partial Application)**: Это самая важная часть. Метод Build создает замыкание. Он берет низкоуровневый транслятор и «приклеивает» его к логике доступа к буферам модели (ConverterForTranslator).
1. **Кэширование (Caching)**: Результат сохраняется в _CachedInputConverter. Теперь у тебя есть «черный ящик», который знает что делать и куда класть, и ему больше не нужны никакие инструкции.

## Этап 2: Использование (Hot Path — выполняется каждый кадр)
Когда ты вызываешь полученный InputConverter внутри метода Predict:

- **Zero Overhead**: Внутри делегата больше нет условий if (algorithm == X), нет switch, нет поиска в словарях.
- **Прямой вызов**: Процессор просто прыгает по адресу функции трансляции. 
 Это позволяет достичь максимального FPS, так как CPU не тратит такты на логику выбора — выбор уже был сделан «две недели назад» на этапе билдера.




# Translators

Берут входные данные (например, List<Mat>) и преобразуют их в формат, 
 который может принять ONNX Runtime (Span<float> или Span<Half>), 
 со всеми необходимыми математическими преобразованиями (нормализация, изменение порядка каналов, цветовая адаптация и т.д.).
 Основная цель — сделать это максимально быстро, так как это часть "горячего пути" (Hot Path) в инференсе.


## Почему это выделено в отдельный слой?
- **Чистота (Isolation)**: Трансляторы — это «чистые функции». 
                _**Они ничего не знают про ONNX, сессии или видеокарты**_. 
                Их можно тестировать обычными Unit-тестами, просто подавая массив чисел на вход.
- **AutoTune**: Благодаря изоляции, система может запустить 10 разных трансляторов на одном кадре, замерить время и выбрать «чемпиона» именно для вашего процессора.
- **Гибкость**: Если завтра появится модель, требующая формат YUV или GrayScale, 
                вам не нужно менять логику работы с ONNX — вы просто дописываете один новый метод-транслятор.

## За что конкретно отвечает Translators:
1. Трансформация данных (Data Transformation):
   - Изменение порядка (Reordering): Перекладывание пикселей из формата HWC (как в OpenCV: Высота-Ширина-Каналы)
     в формат CHW (как обычно ждут нейросети: Каналы-Высота-Ширина).
   - Цветовая адаптация: Переход из BGR (стандарт OpenCV) в RGB.
2. Математическая подготовка (Normalization):
   - Приведение диапазона [0...255] к [0...1] или [-1...1].
   - Применение специфичных коэффициентов (например, Mean/Std для ImageNet).
3. Приведение типов (Type Casting):
   - Конвертация «на лету» из float (FP32) в Half (FP16), чтобы экономить время на отдельном проходе по массиву.
4. Экстремальная производительность (Performance):
   - Unsafe & Pointers: Прямая работа с указателями для обхода проверок границ массивов (bounds checks).
   - SIMD & Векторизация: Использование инструкций процессора для обработки нескольких пикселей за один такт.
   - Zero-Allocation: Трансляторы никогда не создают новые массивы, они работают только с тем буфером, 
      который им «дали в руки». Исключение получение временной памяти через MemoryPool.
